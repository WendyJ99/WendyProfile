移动端 App · 0–1 产品设计

CommitGo（不鸽）

通过家长发布任务、孩子完成任务赚取金币、金币兑换约定奖励的正向激励模式，化解亲子对抗，帮助孩子主动养成好习惯。

时间 2025 春 团队 3 人 我的职责 用户访谈、需求分析、原型、交互设计 产出 高保真原型 + 走查 + 可用性测试

01 / 项目背景与目标

同一款 App，家长与孩子以不同角色使用：家长创建任务与奖励、验收完成情况；孩子接收任务、打卡、赚取金币并兑换奖励。目标是用「约定写清、过程可追溯、兑现可落地」的机制，减少「你耍赖 / 你不守约」的争执，让激励闭环真正跑起来。

02 / 用户洞察

访谈中发现，亲子冲突常集中在「奖励」上：说不清、记不住、标准不统一，最后演变成互相指责。

家长侧 催、记不清、标准不统一
冲突 ⇢
孩子侧 觉得不公、记不住约定
「奖励这件事说不清、记不住、也不统一标准，最后就变成'你耍赖/你不守约'。」

因此产品需要解决三件事：约定写清、过程可记录、兑现可追溯。

03 / 核心机制与用户旅程

机制本质：家长创建任务并设定奖励 → 孩子执行并打卡 → 家长验收 → 金币入账 → 孩子兑换奖励。以下用户旅程图展示亲子双方在各阶段的情绪变化与关键行为。

优化方案 验收通过 / 金币到账时发送推送通知或站内提示，消除孩子等待焦虑，增强「做了就有反应」的正向感受。

约定写清

家长填写完成标准、奖励金币（按难度选择档位或自定义）；孩子确认后任务生效，避免口头约定。

打卡与验收

孩子上传照片/录音/截图打卡；家长通过/驳回（可重提）；通过后金币自动入账。

兑换可追溯

兑换商城由亲子提前共同列好（如看电影、买玩具）；兑换后留记录，避免「答应过没兑现」。

04 / 核心流程：孩子打卡 → 家长验收 → 金币入账

作品集重点呈现的主路径闭环，展示激励机制如何端到端运转。

孩子完成打卡
上传照片/录音/截图 → 家长验收
通过 / 驳回（可重提） → 金币入账 + 弹窗反馈
「太棒了！获得 10 金币」

孩子提交后进入待验收状态流转：

待验收 → ✓ 通过（金币入账） ✕ 驳回（可重提） ⏱ 3 天超时自动通过

3 天超时机制：亲子不一定每天打开 App，超时自动通过避免流程长期卡住。该设定属于机制假设，暂无明确数据支撑。

验证计划：上线后以「待验收停留时长」中位数 < 24 小时为主指标，辅以超时自动通过占比、驳回率等观察迭代。

06 / 关键界面原型

展示家长端任务管理（Mission Board）、奖励商城（Treasure）和验收（Inbox）三个核心界面，以及孩子端任务执行页面。

左：家长端 Mission Board — 创建任务（DRAFT NEW MISSION）、查看各任务进度状态（PENDING / IN PROGRESS / COMPLETED / TODO）
中：家长端 Treasure — 奖励兑换商城（Mystic Cartoon Scroll / Theme Park Ticket 等亲子提前列好）
右：家长端 Inbox — 孩子提交打卡后，家长点击 APPROVE 通过 / Return 驳回，完成验收闭环

07 / 走查与可用性测试

先进行团队内部/同学走查，再找真实用户做可用性测试（有任务、有观察、有记录）。

发现问题 优化方案
部分按钮缺少二次确认，易误触（如删任务） 增加二次确认弹窗
孩子无手机号，无法独立注册 监护人注册并绑定孩子（二维码邀请）
孩子对「待验收」不理解：不知道等多久、接下来干嘛、金币在哪看，容易觉得「做了也没反应」 优化通知：验收通过 / 金币到账的推送或站内提示，让孩子明确看到状态与金币变化

08 / 用户反馈与后续迭代方向

习惯 vs 任务

支持将行为分类为「习惯」（按周期执行，如每日刷牙）与「任务」（单次设置），满足不同激励场景。

扣分项

支持设定扣分规则（如骂人、撒谎扣分），与正向激励形成完整行为约束。
